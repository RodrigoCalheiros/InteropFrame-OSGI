\section{Motivação} \label{sec.motivacao}

A necessidade de soluções em sistemas distribuídos complexos vem substituindo a visão de sistemas distribuídos homogêneos onde aplicações de domínio específico são desenvolvidas usando plataformas e \textit{middleware} projetados especificamente para esse domínio. Soluções tecnológicas independentes têm sido interconectadas para criar estruturas ainda mais ricas, os chamados sistemas de sistemas (SoS). Um dos principais desafios dessas interconexões é a questão da interoperabilidade: a habilidade desses sistemas se conectarem, de trocarem dados e de se comunicarem \cite{blair2011}, \cite{inverardi2010}, \cite{bromberg2011}.

Uma técnica que tem sido amplamente utilizada nos últimos anos no desenvolvimento de plataformas de sistemas distribuídos é a ESBC (Engenharia de Software Baseada em Componentes) \cite{rouvoy2009}. Como consequência do sucesso no uso de componentes de software, vários modelos de componentes diferentes emergiram. OpenCOM~\cite{coulson2008}, Fractal~\cite{bruneton2006}, Spring~\cite{spring2007}, EJB~\cite{ejb2005}, OSGi~\cite{osgi}, CCM~\cite{ccm2002} e SCA~\cite{sca2007} são alguns exemplos destes modelos.

Diversas soluções têm sido criadas para lidar com a interoperabilidade entre sistemas distribuídos desenvolvidos em diferentes modelos de componentes. Uma dessas soluções é o objeto de estudo deste trabalho: O InteropFrame \cite{nascimento2013}.

O InteropFrame é um \textit{middleware} extensível desenvolvido em Java que trata a questão da interoperabilidade entre sistemas desenvolvidos nos modelos de componentes OpenCOM ou Fractal através da geração automática de \textit{proxies} para a comunicação remota. Estes \textit{proxies} atuam como representantes locais a um sistema que repassam as chamadas de métodos para outros computadores dentro de uma rede, seja ela local ou a própria internet. No InteropFrame, as chamadas de métodos podem ser repassadas pela rede utilizando o mecanismo Java RMI (\textit{Remote Method Invocation}) ou \textit{Web Services} do tipo SOAP. Com isso o InteropFrame é capaz de fazer o \textit{binding} (ligação remota) entre partes distribuídas de modo que o utilizador não precise se preocupar com detalhes de comunicação remota nem de aspectos de interoperabilidade entre os modelos de componentes suportados. O InteropFrame é extensível do ponto de vista do desenvolvedor que deseje implementar o suporte a novos modelos de componentes e de \textit{binding} através do desenvolvimento de \textit{plug-ins} à parte.
 
Embora o InteropFrame funcione para os modelos de componentes propostos, ele possui algumas limitações. Uma dessas limitações é que o InteropFrame não é desenvolvido numa plataforma de componentes específica, sendo totalmente desenvolvido em Java. Esse fato acaba limitando o processo de extensibilidade do \textit{middleware}. Outra limitação é a quantidade de modelos de componentes suportados -- apenas Fractal e OpenCOM.

\section{Objetivos do Trabalho}

\subsection{Objetivo Geral}

Superar limitações do InteropFrame e estender o suporte a novos modelos de componentes.

\subsection{Objetivos Específicos}

\begin{itemize}
  \item Estudo do InteropFrame;
  \item Fazer a portabilidade do InteropFrame para o modelo de componentes OSGi como forma de modularizá-lo em \textit{plug-ins}. Dessa forma o suporte à extensibilidade ficará facilitado;
  \item Implementar o suporte ao OSGi como um modelo de componentes interoperável dentro do InteropFrame a fim de comprovar a extensibilidade do \textit{middleware};
	\item Avaliar impactos no desempenho do \textit{middleware} devido à introdução do OSGi;
	\item Realizar melhorias no processo interno de comunicação remota do InteropFrame.
\end{itemize}

\section{Contribuições deste Trabalho}

As principais contribuições deste trabalho são:
\begin{itemize}
	\item Evolução do InteropFrame.
	\item Disponibilização do código-fonte como forma de possibilitar estudos futuros à respeito da interoperabilidade entre sistemas distribuídos desenvolvidos em diferentes modelos de componentes.
\end{itemize}
